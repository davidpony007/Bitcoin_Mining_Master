# 游戏机制功能实现总结

## ✅ 已完成的功能

### 1. 等级系统（Level System）
- **9 个等级**：LV.1 到 LV.9，每级有不同的积分要求和挖矿速度倍数
- **积分机制**：统一的积分系统，通过签到、广告、邀请等方式获取
- **挖矿加速**：等级越高，挖矿速度倍数越高（1.0x ~ 3.0x）
- **排行榜**：按等级和积分排序的全球排行榜

**API 端点**：
- `GET /api/level/info?user_id=xxx` - 查询用户等级信息
- `GET /api/level/config` - 获取所有等级配置
- `GET /api/level/leaderboard?limit=100` - 等级排行榜
- `GET /api/level/mining-speed?user_id=xxx` - 计算挖矿速度

### 2. 签到系统（Check-in System）
- **每日签到**：用户每天可签到一次获得积分
- **连续签到奖励**：
  - 1天: +4积分
  - 3天: +15积分
  - 7天: +30积分
  - 15天: +30积分
  - 30天: +60积分
- **每日加成**：签到后激活 1.36x 挖矿倍数，持续 2 小时

**API 端点**：
- `POST /api/checkin` - 执行签到
- `GET /api/checkin/status?user_id=xxx` - 查询签到状态
- `GET /api/checkin/history?user_id=xxx&days=30` - 签到历史
- `GET /api/checkin/statistics?user_id=xxx` - 签到统计

### 3. 广告系统（Advertisement System）
- **观看广告奖励**：每观看一个完整广告获得 +20 积分
- **推荐人奖励**：被推荐人每观看 10 个广告，推荐人获得 +1 积分
- **每日统计**：记录用户每天观看广告的次数
- **观看记录**：完整的广告观看历史和时长记录

**API 端点**：
- `POST /api/ad/watch` - 记录广告观看
- `GET /api/ad/count?user_id=xxx` - 今日广告观看次数
- `GET /api/ad/statistics?user_id=xxx` - 广告观看统计
- `GET /api/ad/history?user_id=xxx&page=1&limit=20` - 广告观看历史
- `GET /api/ad/referral-progress?user_id=xxx` - 推荐人的广告收益进度

### 4. 邀请系统（Invitation System）
- **邀请新用户**：每邀请一个新用户获得 +6 积分
- **里程碑奖励**：
  - 邀请 5 人: +20 积分
  - 邀请 10 人: +30 积分
- **广告连带奖励**：被邀请人观看广告，邀请人也获得奖励
- **邀请排行榜**：按邀请人数排序

**API 端点**：
- `GET /api/invitation/progress?user_id=xxx` - 邀请进度
- `GET /api/invitation/referrals?user_id=xxx&page=1&limit=20` - 被邀请人列表
- `GET /api/invitation/statistics?user_id=xxx` - 邀请统计
- `GET /api/invitation/leaderboard?limit=100` - 邀请排行榜

### 5. 积分系统（Points System）
- **统一积分交易**：所有积分变动都通过存储过程记录
- **完整审计日志**：`points_transaction` 表记录每一笔积分变动
- **自动等级升级**：积分达到阈值时自动升级
- **手动调整**：管理员可以手动增减用户积分

**API 端点**：
- `GET /api/points/history?user_id=xxx&page=1&limit=20` - 积分交易历史
- `POST /api/points/add` - 手动增加积分（管理员）

## 🗄️ 数据库设计

### 新增表（7张）

1. **level_config** - 等级配置表
   - 9 个等级的配置（积分要求、挖矿倍数）

2. **check_in_reward_config** - 签到奖励配置表
   - 5 个连续签到档位的奖励配置

3. **user_check_in** - 用户签到记录表
   - 记录每次签到的日期、连续天数

4. **ad_watch_record** - 广告观看记录表
   - 记录广告类型、观看时长、是否完成

5. **points_transaction** - 积分交易记录表
   - 所有积分变动的审计日志

6. **invitation_reward_progress** - 邀请奖励进度表
   - 里程碑奖励的领取状态

7. **referral_ad_watch_count** - 推荐人广告计数表
   - 统计被推荐人的广告观看次数

### 扩展表

**user_information** 新增字段：
- `user_level` - 用户等级
- `user_points` - 用户积分
- `mining_speed_multiplier` - 挖矿速度倍数

### 存储过程

**sp_add_user_points** - 原子化的积分增加操作
- 输入：用户ID、积分、原因、类型等
- 输出：操作前后的积分、等级、是否升级
- 自动：计算新等级、更新倍数、记录交易日志

### 视图

**v_user_full_info** - 用户完整信息视图
- 合并 user_information 和 level_config
- 计算当前等级进度百分比
- 计算距离下一级所需积分

## ⚙️ 后端服务

### 服务层（4个服务）

1. **LevelService** - 等级和积分管理
   - 等级配置加载
   - 积分计算和等级判定
   - 挖矿速度计算（等级倍数 × 每日加成）
   - 排行榜查询

2. **CheckInService** - 签到系统
   - 签到奖励配置加载
   - 连续签到天数计算
   - 每日加成激活（1.36x，2小时）
   - 签到历史和统计

3. **AdService** - 广告系统
   - 广告观看记录
   - 积分奖励发放（+20）
   - 推荐人奖励（每10个广告 +1）
   - 每日观看次数统计

4. **InvitationRewardService** - 邀请奖励系统
   - 新用户注册奖励（+6）
   - 里程碑检测和发放（+20, +30）
   - 邀请进度跟踪
   - 邀请排行榜

### Redis 缓存策略

**缓存类型**：
- **用户等级**：TTL 24小时，热数据
- **签到状态**：TTL 48小时，每日更新
- **每日广告计数**：自动在当天结束时过期
- **推荐人广告计数**：TTL 30天
- **邀请进度**：长期缓存，手动失效
- **每日加成用户**：Sorted Set，按过期时间排序

**缓存更新策略**：
- 写入时同时更新 MySQL 和 Redis
- 读取时优先从 Redis，未命中则查询 MySQL
- 数据变更时立即删除相关缓存

### 定时任务（6个任务）

1. **每日加成过期清理** - 每分钟执行
   - 清理 Redis 中已过期的每日加成用户

2. **每日广告计数重置** - 每天凌晨 0:00
   - Redis 自动过期处理

3. **签到数据同步** - 每小时执行
   - 验证 Redis 和 MySQL 的签到数据一致性

4. **等级缓存预热** - 每天凌晨 3:00
   - 为活跃用户预加载等级缓存

5. **邀请进度同步** - 每 6 小时执行
   - 验证邀请进度数据一致性

6. **推荐人广告计数同步** - 每小时执行
   - 验证推荐人广告计数一致性

## 🔄 业务流程

### 新用户注册流程

```
1. 用户A安装APP，自动注册
   ↓
2. 生成user_id和invitation_code
   ↓
3. 创建user_information记录
   ↓
4. 初始化user_status记录
   ↓
5. user_level=1, user_points=0
   ↓
6. 返回用户信息和邀请码
```

### 用户B通过邀请码注册流程

```
1. 用户B输入邀请码注册
   ↓
2. 验证邀请码有效性
   ↓
3. 创建invitation_relationship记录
   ↓
4. 调用InvitationRewardService.handleNewReferral()
   ↓
5. 推荐人A获得+6积分（基础奖励）
   ↓
6. 检查推荐人A的邀请总数
   ↓
7. 如果达到5人，再奖励+20积分
   ↓
8. 如果达到10人，再奖励+30积分
   ↓
9. 记录到points_transaction表
   ↓
10. 更新推荐人A的等级（如果积分足够）
   ↓
11. 清除推荐人A的所有相关缓存
```

### 每日签到流程

```
1. 用户点击签到按钮
   ↓
2. 检查今天是否已签到（Redis缓存）
   ↓
3. 查询上次签到日期（MySQL）
   ↓
4. 计算连续签到天数
   ↓
5. 根据连续天数确定奖励档位
   ↓
6. 插入user_check_in记录
   ↓
7. 通过存储过程增加积分
   ↓
8. 激活1.36x挖矿加成，2小时有效
   ↓
9. 更新Redis缓存（签到状态、等级信息）
   ↓
10. 添加到每日加成Sorted Set
   ↓
11. 返回签到结果和奖励信息
```

### 观看广告流程

```
1. 用户观看广告30秒
   ↓
2. 记录ad_watch_record（type, duration, completed）
   ↓
3. 如果观看完成，奖励用户+20积分
   ↓
4. 增加今日广告计数（Redis）
   ↓
5. 查询用户的推荐人
   ↓
6. 如果有推荐人，增加推荐关系的广告计数
   ↓
7. 检查是否达到10的倍数
   ↓
8. 如果是，奖励推荐人+1积分
   ↓
9. 更新推荐人的rewarded_count字段
   ↓
10. 清除用户和推荐人的所有缓存
   ↓
11. 返回观看结果和奖励信息
```

## 🎯 性能指标

### 预期性能

- **签到接口响应时间**: < 200ms
- **广告观看记录**: < 300ms
- **等级查询（缓存命中）**: < 50ms
- **等级查询（缓存未命中）**: < 150ms
- **排行榜查询**: < 500ms

### 并发支持

- **签到系统**: 支持 1000+ 并发
- **广告系统**: 支持 500+ 并发
- **等级查询**: 支持 5000+ 并发（Redis缓存）

### 存储预估

假设 10,000 活跃用户：

| 表名 | 每日新增记录 | 月增长 | 年增长 |
|------|--------------|--------|--------|
| user_check_in | ~7,000 | 210K | 2.5M |
| ad_watch_record | ~50,000 | 1.5M | 18M |
| points_transaction | ~70,000 | 2.1M | 25M |
| invitation_relationship | ~100 | 3K | 36K |

**建议**：
- 广告观看记录每季度归档一次

- Redis 内存预留 256MB-512MB
